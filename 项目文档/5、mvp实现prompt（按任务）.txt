Mare MVP 实现 Prompt（按任务）

一、说明

本文件用于沉淀 Mare 项目的任务化实现 prompt 风格，便于后续继续按阶段推进。

当前项目事实如下：

1. 当前项目名称为 Mare。
2. 当前项目是桌面客户端程序，不是普通网页程序。
3. 当前架构是 Tauri 负责客户端壳层与前端承载，Go 负责后端。
4. 原 prompt 中提到 Rust 后端的地方，统一调整为 Go 后端。

二、总投喂原则

1. 优先按 A1 -> D5 打通最小闭环的顺序理解项目推进。
2. 结合当前代码事实执行，不重复做已经完成的基础骨架。
3. 如任务已经部分完成，则 prompt 的使用方式应从“从零实现”调整为“在现有代码上补齐、重构或完善”。

三、阶段化 prompt 使用说明

A1 创建桌面端工程骨架
使用口径：
为 Mare 桌面客户端维护和整理 Tauri + React + TypeScript 前端骨架，保持 pages、components、layouts、hooks、services、types、styles 结构清晰，不引入与业务无关的演示代码。

A2 创建 Go 后台核心模块骨架
使用口径：
为 Mare 创建或整理 Go 后端核心模块骨架，模块至少覆盖 catalog、sync、import、search、connectors、tasks、config，并提供统一 App 总入口，不把业务都堆在 main.go 中。

A3 建立本地 SQLite 数据库初始化机制
使用口径：
实现或完善本地 SQLite 初始化和 migration 机制，保证首次启动自动建库，再次启动不重复执行已完成 migration。

B1 到 B4
使用口径：
围绕 storage_endpoints、assets、replicas、replica_versions、tasks 建立或完善模型、表结构、DAO、测试和最小服务层。

C1 到 C6
使用口径：
围绕统一 Connector 接口，持续补齐 Local、QNAP、115、Removable 的连接能力与测试。

D1 到 D5
使用口径：
围绕路径归一化、扫描、资产归并、全量扫描、单端点重扫建立稳定 Catalog 链路。

E1 到 E4
使用口径：
围绕 Mare 桌面客户端统一视觉与资产浏览体验，维护主布局、资产列表、筛选排序和资产详情，不参考旧界面效果，以当前产品视觉方向为准。

F1 到 F4
使用口径：
围绕图片缩略图、视频封面、音频元数据和图片/视频/音频预览持续补齐派生媒体与前端展示闭环。

G1 到 H3
使用口径：
围绕副本状态聚合、差异分析、恢复、删除和资产可见性规则继续推进核心资产管理能力。

I1 到 I5
使用口径：
围绕可移动设备识别、角色选择、导入源浏览、导入规则与导入执行器继续推进导入中心。

J1 到 J6
使用口径：
围绕全文搜索、筛选联动、转写、图片语义搜索、视频语义搜索与统一搜索页推进智能检索能力。

K1 到 K5
使用口径：
围绕配置导入导出、端点重绑定和换机恢复推进可迁移性。

L1 到 L3
使用口径：
围绕后台任务中心、任务重试和结构化日志推进可维护性。

四、建议统一 prompt 模板

后续若要继续按任务推进，建议统一使用如下模板：

任务名称：
填写当前任务编号和标题，例如 F2 实现视频封面提取。

项目背景：
1. 项目名称是 Mare。
2. 当前是 Tauri 桌面客户端 + Go 后端架构。
3. 事实以当前仓库代码为准，不以历史文档为准。

任务目标：
清楚描述这次要实现或完善的功能闭环。

当前代码前提：
1. 说明当前已有哪些现成模块可复用。
2. 说明哪些旧实现不能直接假设。

实现要求：
1. 列出本次必须完成的能力。
2. 列出接口、页面、服务、数据库、测试等交付物。

约束：
1. 不要破坏现有桌面端结构。
2. 不要把客户端写成普通网页。
3. 不要忽略 Go 后端既有模块。
4. 尽量保持代码清晰、可维护、可测试。

验收标准：
1. 列出用户最终可以验证的行为。
2. 如需依赖 FFmpeg、SQLite、Air 等环境，明确说明。

五、示例修正版 prompt

示例一：继续完善 Go 后端任务

你要为 Mare 项目完善 Go 后端的某项能力。

背景：
- 项目是桌面客户端程序，不是单纯网页程序。
- 前端使用 Tauri + React + TypeScript。
- 后端使用 Go。
- 当前事实以仓库代码为准。

任务要求：
- 在现有 backend 模块基础上实现或完善指定能力。
- 保持 HTTP 接口、Catalog、Store、Connector 边界清晰。
- 优先复用已有 SQLite、任务表和媒体处理能力。

约束：
- 不要再按 Rust 后端方案设计新主干。
- 不要忽略现有表结构和服务层。

交付物：
- Go 代码实现
- 必要的测试
- 与前端或接口的最小接入

示例二：继续完善桌面客户端页面

你要为 Mare 桌面客户端继续实现某个页面或交互。

背景：
- 当前客户端由 Tauri 承载。
- 前端使用 React + TypeScript。
- 设计方向要求简洁、克制、现代、桌面客户端化。

任务要求：
- 在现有 AppShell、页面路由、主题系统和服务接入基础上继续实现指定页面。
- 保持浅色与深色主题结构一致。
- 如需读取数据，优先接当前 Go 后端接口。

约束：
- 不要把页面做成传统后台表格堆砌风格。
- 不要写成仅适合网页的交互。

六、结论

本文件用于帮助后续继续沿用任务化推进方式。
真正执行时必须遵循以下优先级：

1. 当前代码事实
2. 当前架构事实：Tauri + Go
3. 当前 MVP 任务顺序和范围
