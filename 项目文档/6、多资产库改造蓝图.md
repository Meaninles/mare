# Mare 多资产库改造蓝图

## 1. 文档目的

本文档用于为 Mare 当前项目提供一份正式的“多资产库改造蓝图”，作为后续所有实现任务的统一基线。

本蓝图只处理架构方向、边界划分、模块职责、迁移路径和阶段目标，不直接展开代码实现细节。

本蓝图必须同时满足以下目标：

1. 让系统回归“以资产库为核心”的产品模型。
2. 明确“应用壳数据”和“资产库数据”的职责边界。
3. 指导 `backend`、`src`、`src-tauri` 三层按同一方向演进。
4. 为后续任务 2 到 9 提供稳定前提，避免局部修补再次回到全局单体状态。

## 2. 总体结论

Mare 当前实现虽然在产品描述上已经使用了 Catalog 的术语，但真实架构仍然是“单体全局 Catalog”。

当前系统必须立刻停止继续沿“全局唯一库 + 全局唯一状态”方向扩展，改为以下目标结构：

`系统 -> 资产库 -> 存储节点 -> 资产 -> 副本`

核心架构决策如下：

1. 一个资产库对应一个独立的 SQLite 数据库文件，建议使用 `.maredb` 作为资产库文件后缀。
2. 不采用“在现有所有业务表中加 `workspace_id`”的多租户方案。
3. 采用“两层数据模型”：
   - 应用壳数据库：保存全局状态。
   - 资产库数据库：保存库内 Catalog 数据。
4. 后端从“启动即绑定唯一数据库”改为“启动后空载，按需挂载当前资产库”。
5. 前端从“启动即进入资产页”改为“先进入 Welcome Screen，再打开或创建资产库”。
6. 第一阶段只支持“单窗口单资产库”，不做并行多库运行。

## 3. 当前现状与结构性问题

### 3.1 当前后端是单体全局 Catalog

当前 Go 后端在配置层默认使用单一数据库路径：

- `backend/internal/config/config.go`
- `CATALOG_DB_PATH` 默认值为 `./data/mam.db`

当前应用启动时直接创建这个唯一 Store：

- `backend/internal/app/app.go`
- `store.NewSQLiteStore(cfg.CatalogDBPath)`

这意味着：

1. 进程生命周期和唯一数据库文件强绑定。
2. 所有端点、资产、任务、扫描、副本都落在同一个 Catalog 里。
3. 系统不存在“当前资产库”的显式概念。

### 3.2 当前前端启动流程错误

当前前端路由默认直接跳转资产页：

- `src/App.tsx`
- 默认 index 路由直接 `Navigate to="/library"`

这会导致产品入口被错误定义为：

`打开软件 -> 直接看全局资产`

而不是：

`打开软件 -> 选择或创建资产库 -> 进入该资产库`

### 3.3 当前前端请求与缓存也是全局单例

当前 Catalog 访问模式仍然是全局单例：

1. `src/services/catalog.ts` 写死默认后端地址 `http://127.0.0.1:8080`
2. `src/hooks/useCatalog.ts` 使用全局 query key，如：
   - `catalog-assets`
   - `catalog-endpoints`
   - `catalog-tasks`

这意味着：

1. 查询结果没有“资产库上下文”。
2. 一旦未来支持切库，缓存会互相污染。
3. 未打开资产库前，前端也无法正确表达“当前处于空载状态”。

### 3.4 当前数据模型虽然接近正确，但被放在了错误层级

当前库内表模型已经具备较好的资产建模基础：

- `storage_endpoints`
- `assets`
- `replicas`
- `replica_versions`
- `tasks`

这些表本质上适合作为“单个资产库内部的数据模型”。

问题不在表本身，而在于它们被错误地当成了“整个应用唯一全局状态”。

### 3.5 当前存在跨项目错误合并风险

当前 `assets.logical_path_key` 仍然是全局唯一：

- `backend/internal/store/migrations/0001_core_tables.sql`
- `logical_path_key TEXT NOT NULL UNIQUE`

当前扫描归并逻辑也是按 `logical_path_key` 直接查找资产：

- `backend/internal/store/assets.go`
- `GetAssetByLogicalPathKey`
- `backend/internal/catalog/service.go`

这意味着不同项目、不同拍摄任务、不同资产库中，只要逻辑路径碰巧相同，就可能在同一个全局库中被错误合并。

这对未来“大量照片 + 大量视频 + 多项目长期归档”场景是不可接受的。

### 3.6 当前应用壳数据库已经存在，但职责尚未发挥

`src-tauri` 已经有一套独立的应用数据库：

- `src-tauri/src/database.rs`
- 当前路径位于 Tauri app data 目录下的 `catalog/mam.sqlite3`

当前它只保存非常轻量的 `app_metadata`，但它已经天然适合作为：

1. 最近打开资产库列表
2. 当前活动资产库
3. 应用级偏好
4. 资产库注册记录

的承载层。

因此，Mare 并不是要“凭空新造第二套数据库机制”，而是要把已经存在但未真正使用的应用壳数据库扶正。

### 3.7 当前设置备份与 Catalog 快照边界混杂

`backend/internal/catalog/backup.go` 当前同时处理：

1. 应用偏好
2. 存储端配置
3. 导入规则
4. Catalog 快照

这在“单资产库时代”还能工作，但切到多资产库后会产生明显问题：

1. 应用设置和资产库归档不是一回事。
2. 资产库应当首先被视为独立文件，而不是附属 JSON 备份对象。
3. “换机恢复应用”与“迁移某一个资产库”必须拆成两类能力。

## 4. 目标产品模型

### 4.1 核心层级

目标层级固定为：

`系统 -> 资产库 -> 存储节点 -> 资产 -> 副本`

具体解释如下：

1. 系统
   - 指应用本身。
   - 负责欢迎页、最近打开、应用偏好、资产库注册。
2. 资产库
   - 是用户真正操作的顶层业务单元。
   - 一个资产库对应一组独立的配置、索引、任务和视图。
3. 存储节点
   - 是某个资产库下绑定的物理或逻辑存储端。
   - 例如本地 SSD、QNAP、115、移动硬盘。
4. 资产
   - 是 Catalog 中的核心对象。
   - 代表用户真正管理的文件资产，而不是某个磁盘上的某个路径。
5. 副本
   - 描述某个资产在某个存储节点上的一个物理存在形式。
   - 例如 RAW 原片、副本、预览素材、恢复来源等。

### 4.2 产品入口模型

启动软件后的正确流程必须是：

1. 打开 Welcome Screen
2. 选择“新建资产库”或“打开已有资产库”
3. 如果有最近使用记录，则展示最近资产库列表
4. 打开具体资产库后，才进入库内导航与资产视图

不允许再出现“未选择资产库就直接进入资产页”的流程。

### 4.3 库内导航模型

进入资产库后，导航必须围绕“当前资产库”组织，而不是围绕“全局系统”组织。

目标导航结构如下：

1. Assets
   - 统一资产瀑布流或列表视图
2. Collections
   - 逻辑分类、智能集合、标签、评分、颜色标记
3. Storage Nodes
   - 当前资产库下的存储节点配置与状态
4. Import / Ingest
   - 导入源识别、规则执行、入库流程
5. Sync / Replicas
   - 副本状态、补齐、恢复、同步任务
6. Tasks
   - 当前资产库的任务中心
7. Settings
   - 应用级设置，不属于库内导航主体

第一阶段可允许 `Collections` 先做占位页，但路由和信息架构必须预留。

## 5. 目标数据边界

### 5.1 总体原则

所有数据必须先回答一个问题：

“这条数据是属于整个应用，还是只属于某一个资产库？”

如果答案是“只属于某一个资产库”，就必须进入资产库数据库，而不是继续放在全局库里。

### 5.2 数据边界表

| 数据域 | 存放位置 | 说明 |
| --- | --- | --- |
| 最近打开资产库 | 应用壳数据库 | 应用级状态 |
| 当前活动资产库 | 应用壳数据库 | 应用级状态 |
| 应用主题、窗口偏好 | 应用壳数据库 | 应用级状态 |
| 资产库注册记录 | 应用壳数据库 | 包含名称、路径、最近打开时间等 |
| 存储节点定义 | 资产库数据库 | 每个资产库单独维护 |
| 资产、预览、媒体元数据 | 资产库数据库 | 绝对属于库内数据 |
| 副本、副本版本 | 资产库数据库 | 绝对属于库内数据 |
| 导入规则 | 资产库数据库 | 只对当前资产库生效 |
| 扫描任务、同步任务、恢复任务 | 资产库数据库 | 只对当前资产库生效 |
| 集合、标签、评分、颜色标记 | 资产库数据库 | 库内语义组织能力 |
| 敏感凭据实体 | 本机密钥存储或应用壳数据库的安全区 | 库文件中只保留引用，不直接保留秘密值 |
| 预览缓存文件 | 本机 app data 按库分桶目录 | 不建议直接跟随库文件目录 |
| 后端日志 | 本机 app data 按库分桶目录 | 不建议写在资产库目录旁 |

### 5.3 关于凭据

当前 `connection_config` 中已经出现敏感凭据直接落库的情况，例如 115 token。

目标方案应改为：

1. 资产库中保留节点定义、根路径、连接策略、`credential_ref`
2. 真正的密钥材料放到本机安全存储
3. 迁移阶段允许保留兼容逻辑，但目标方向必须明确为“库内配置可携带，秘密值不可裸带”

## 6. 目标物理存储结构

### 6.1 应用壳数据库

应用壳数据库继续由 `src-tauri` 管理，但职责要扩展。

建议目标目录：

`<app_data>/app-shell/app.sqlite3`

说明：

1. 当前 `src-tauri` 已有数据库初始化能力，可直接复用。
2. 目录名和文件名应逐步从 `catalog/mam.sqlite3` 迁到更准确的应用壳命名。
3. 第一阶段允许保留旧路径兼容，但职责必须转为“应用壳库”。

### 6.2 资产库数据库

每个资产库一个独立文件。

建议文件示例：

`D:\Mare Libraries\2026新藏线.maredb`

资产库数据库中保存：

1. 存储节点
2. 资产
3. 副本
4. 副本版本
5. 导入规则
6. 任务
7. 预览记录
8. 音频元数据
9. 后续集合与标签

### 6.3 库级缓存与日志

第一阶段不建议把缓存和日志写在 `.maredb` 同目录。

建议本机路径：

1. 缓存：`<app_data>/libraries/<library_id>/cache/media`
2. 日志：`<app_data>/libraries/<library_id>/logs/backend.log`
3. 临时物化目录：`<app_data>/libraries/<library_id>/temp`

原因：

1. 避免对移动盘、NAS、只读目录产生额外写入依赖
2. 避免缓存文件跟着资产库文件一起被同步或归档
3. 减少跨平台目录权限问题

## 7. 目标后端架构

### 7.1 目标运行模型

当前后端模式：

`进程启动 -> 打开唯一 mam.db -> 创建唯一 Store -> 创建唯一 Catalog Service`

目标后端模式：

`进程启动 -> 空载状态 -> 接收打开资产库请求 -> 动态创建当前 Library Runtime`

### 7.2 目标状态机

后端运行时应至少支持以下状态：

1. Unloaded
   - 已启动，但未打开资产库
2. Loading
   - 正在创建或打开资产库
3. Loaded
   - 当前资产库已挂载，可访问库内 API
4. Closing
   - 正在关闭当前资产库
5. Error
   - 当前库打开失败或运行时出错

### 7.3 目标核心对象

建议新增 `LibraryRuntime` 或 `LibrarySessionManager` 作为后端核心对象，负责：

1. 当前资产库路径
2. 当前库的 `Store`
3. 当前库的 `Catalog Service`
4. 当前库的缓存目录
5. 当前库的日志上下文

`App` 本身不再长期持有固定 `store`，而是持有“可切换的当前库运行时”。

### 7.4 目标 API 分层

后端 API 应明确分为两类：

#### A. 应用壳 / 资产库会话 API

用于库文件生命周期控制，例如：

1. 创建资产库
2. 打开资产库
3. 关闭资产库
4. 获取当前资产库状态
5. 获取当前资产库元信息

#### B. 库内 Catalog API

用于已打开资产库后的业务操作，例如：

1. 资产列表
2. 存储节点管理
3. 扫描
4. 导入
5. 同步
6. 恢复
7. 任务

规则如下：

1. 当未打开资产库时，Catalog API 必须返回明确错误，如 `409 Library Not Open`
2. 前端必须依赖这个状态做页面路由保护

### 7.5 目标 Store 策略

当前 `backend/internal/store` 中的绝大多数方法可以保留，但调用方式要从：

`全局唯一 Store`

改为：

`当前 LibraryRuntime 所属 Store`

也就是说：

1. `store` 包是可复用的
2. `catalog` 包大部分资产和副本逻辑也是可复用的
3. 真正要重构的是它们的装配方式和生命周期

### 7.6 目标迁移策略

资产库数据库继续沿用 migration 机制。

但 migration 的作用域要改成：

`对某个新建或打开的资产库文件执行库内 schema 初始化`

而不是：

`对应用唯一主库执行初始化`

## 8. 目标前端架构

### 8.1 前端状态分层

前端状态必须拆成两层：

1. App Shell State
   - 应用是否已启动
   - 最近资产库列表
   - 当前活动资产库
   - 应用主题与偏好
2. Library State
   - 当前库的资产、节点、任务、导入状态、同步状态

### 8.2 目标路由模型

建议目标路由如下：

1. `/welcome`
2. `/libraries/new`
3. `/libraries/:libraryId/assets`
4. `/libraries/:libraryId/collections`
5. `/libraries/:libraryId/storage`
6. `/libraries/:libraryId/import`
7. `/libraries/:libraryId/sync`
8. `/libraries/:libraryId/tasks`
9. `/settings`

说明：

1. `libraryId` 在路由层显式存在，便于前端状态校验与刷新恢复
2. 当前活动资产库与路由参数必须一致
3. 未打开资产库时禁止进入任何 `/libraries/:libraryId/*` 页面

### 8.3 目标页面入口规则

前端入口规则必须调整为：

1. 如果无当前活动资产库，默认进入 `/welcome`
2. 如果有当前活动资产库，才跳入该库的 Assets 页面
3. 如果路由参数指向的库不是当前已挂载库，则需要先切换或重新挂载

### 8.4 目标 Query Key 规范

所有库内 Query Key 必须带 `libraryId`。

例如：

1. `["library", libraryId, "assets", limit]`
2. `["library", libraryId, "endpoints"]`
3. `["library", libraryId, "tasks", limit]`
4. `["library", libraryId, "sync-overview"]`

这样可以保证：

1. 切库不会串缓存
2. 失效策略只作用于当前资产库
3. 多库历史缓存可被明确隔离

### 8.5 目标导航结构

现有 `AppShell`、`SidebarNav` 可以保留作为视觉骨架，但信息架构必须重做。

重构重点如下：

1. 顶部和侧边栏显示当前资产库名称
2. `StoragePage` 改名理解为“当前库的 Storage Nodes”
3. `LibraryPage` 变成“当前库的 Assets”
4. `SettingsPage` 从库内能力中抽离出应用级设置
5. Welcome Screen 不使用当前库布局骨架

## 9. src-tauri 目标职责

### 9.1 应用壳数据承载层

`src-tauri` 应成为应用壳状态的真实承载者，负责：

1. 管理应用壳数据库
2. 提供资产库注册、最近打开、当前活动库命令
3. 提供文件选择器能力
4. 提供应用启动 bootstrap

### 9.2 目标 Tauri 命令

建议在 `src-tauri` 中新增或扩展命令：

1. `get_shell_bootstrap`
2. `list_libraries`
3. `create_library_record`
4. `register_existing_library`
5. `set_active_library`
6. `clear_active_library`
7. `open_library_picker`

### 9.3 与 Go 后端的协作边界

职责拆分如下：

1. `src-tauri`
   - 管理应用壳数据库
   - 做文件选择、最近记录、当前库记录
2. Go 后端
   - 创建资产库文件
   - 打开资产库文件
   - 提供库内 Catalog 能力

前端执行开库流程时，应先拿到文件路径，再调用后端挂载当前库，成功后更新应用壳当前活动库记录。

## 10. 现有模块保留、重构与新增清单

### 10.1 可以保留的部分

以下内容大体正确，应尽量保留：

1. `backend/internal/store`
   - 作为库内数据访问层保留
2. `backend/internal/catalog`
   - 资产、副本、扫描归并、媒体派生逻辑保留
3. `backend/internal/connectors`
   - 统一存储接入抽象保留
4. `src/components/media`
   - 预览组件保留
5. `src/pages/LibraryPage`
   - 作为 Assets 页的主体内容可继续演进
6. `src/pages/StoragePage`
   - 作为库内 Storage Nodes 页面继续演进
7. `src-tauri` 数据库初始化能力
   - 直接转为应用壳数据库能力

### 10.2 必须重构的部分

以下内容必须重构，不能继续沿用现状：

1. `backend/internal/config/config.go`
   - 不能再把 `CATALOG_DB_PATH` 当唯一业务主库
2. `backend/internal/app/app.go`
   - 不能在启动时固定构造唯一 Store 和唯一 Catalog Service
3. `backend/internal/httpserver`
   - 必须增加资产库会话 API
4. `src/App.tsx`
   - 默认路由不能再直接进入 `/library`
5. `src/layouts/AppShell.tsx`
   - 需要区分欢迎态和库内态
6. `src/components/SidebarNav.tsx`
   - 需要改成库内导航结构
7. `src/hooks/useCatalog.ts`
   - Query Key 必须绑定 `libraryId`
8. `src/services/catalog.ts`
   - 需要引入“当前库上下文”和未开库状态处理
9. `backend/internal/catalog/backup.go`
   - 要拆分“应用设置备份”和“资产库迁移/归档”

### 10.3 必须新增的部分

以下能力是当前缺失、但多资产库架构必需新增的：

1. 资产库注册表
2. 最近打开列表
3. 当前活动资产库上下文
4. Welcome Screen
5. 创建 / 打开 / 关闭资产库 API
6. `LibraryRuntime` 或等效对象
7. 旧 `mam.db` 迁移工具或迁移流程
8. 应用壳与库内状态分层 bootstrap

## 11. 迁移顺序

后续改造建议严格按以下顺序推进。

### 阶段 0：蓝图冻结

目标：

1. 明确“一个资产库 = 一个独立数据库文件”
2. 明确应用壳与资产库的边界
3. 明确第一阶段只做单窗口单资产库

交付物：

1. 本蓝图文档

### 阶段 1：建立应用壳数据库中的资产库注册能力

目标：

1. 应用侧能记录资产库
2. 支持最近打开和当前活动资产库
3. 先把“库”的概念从 UI 和应用壳层立起来

### 阶段 2：后端改造为按需挂载资产库

目标：

1. 后端可以空载启动
2. 可以动态打开和关闭当前资产库
3. 库内 API 在无当前库时明确拒绝访问

### 阶段 3：重做前端启动流程和 Welcome Screen

目标：

1. 不再默认直接进入资产页
2. 新建资产库、打开已有资产库、最近打开列表可用

### 阶段 4：把前端缓存与请求全部切到“当前库上下文”

目标：

1. 所有 Query Key 带 `libraryId`
2. 切库不会串状态

### 阶段 5：重构库内导航

目标：

1. 资产、集合、存储节点、导入、同步、任务成为清晰的库内结构

### 阶段 6：把存储、导入、扫描、同步、任务全部库内化

目标：

1. 所有业务操作都明确作用于当前资产库

### 阶段 7：旧库迁移

目标：

1. 把当前 `backend/data/mam.db` 迁移为一个默认资产库
2. 保留可验证与可回退路径

## 12. 第一阶段约束

为了控制复杂度，第一阶段明确不做以下内容：

1. 不做单进程多资产库并发运行
2. 不做多窗口多库同时打开
3. 不做复杂权限系统
4. 不做远程多人协同
5. `Collections` 可以先只搭信息架构，不要求全部功能一次完成

## 13. 完成判定标准

当以下条件全部满足时，才能认为多资产库架构完成了第一阶段落地：

1. 全新启动应用时进入 Welcome Screen，而不是直接进入资产页
2. 可以创建多个资产库文件并分别打开
3. 每个资产库拥有自己的存储节点配置
4. 资产、任务、导入规则、同步状态不会跨库污染
5. 切换资产库后，前端缓存不会串数据
6. 未打开资产库时访问库内 API 会得到明确拒绝
7. 旧 `mam.db` 可以迁移成默认资产库
