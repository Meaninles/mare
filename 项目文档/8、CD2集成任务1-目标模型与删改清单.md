# 8、CD2集成任务1：目标模型与删改清单

## 1. 任务目标

本任务用于冻结 CloudDrive2（下文简称 `CD2`）集成改造的目标模型、系统边界和旧能力淘汰范围，作为后续任务 2-8 的唯一实施基线。

本任务完成后，后续开发必须遵守以下前提：

1. 本项目内所有“网盘文件扫描、浏览、管理、上传、下载、任务监控”等能力，统一改由 `CD2 gRPC API` 执行。
2. 现有 `115 + AList + aria2 + Python sidecar + Tauri p115 sidecar` 的网盘执行链路全部淘汰。
3. 现有网盘能力中，**仅允许保留“登录获取 cookie”这一引导能力**，且该能力只用于将账号导入到 CD2，不再直接承担文件操作。
4. 当前处于开发阶段，**不做旧网盘实现复用，不做旧数据模型兼容优先**，以最优架构为第一目标。

---

## 2. 改造范围定义

### 2.1 本次替换的对象

以下能力全部纳入 CD2 化改造范围：

1. 网盘接入与登录
2. 网盘目录浏览、文件搜索、文件详情读取
3. 创建目录、重命名、移动、复制、删除
4. 文件上传、下载、下载直链获取
5. 网盘任务状态获取、暂停、恢复、取消
6. 资产扫描、目录同步、网盘副本恢复
7. 网盘配置管理、云账号管理

### 2.2 不在本次删除范围内的对象

以下模块不属于“网盘能力”，不在本次淘汰范围：

1. 本地目录接入
2. QNAP / SMB 接入
3. 可移动设备接入
4. 媒体处理、搜索、资产库核心模型

### 2.3 唯一允许保留的旧网盘能力

允许保留：

1. 115 登录拿 cookie 的能力

保留后的定位：

1. 它只作为“导入 CD2 云账号”的前置工具
2. 它不再提供 `list/stat/upload/download/delete/move/rename/mkdir` 等任何文件能力
3. 它不再出现在正式的网盘执行链路中

---

## 3. 目标架构原则

### 3.1 执行原则

1. **CD2 gRPC 是唯一网盘执行通道**
2. **CD2 挂载点不是主执行通道**
3. **路径 API 优先于挂载盘 API**
4. **任务中心展示的是“CD2 任务镜像”，不是自研上传引擎**
5. **CD2 内部负责网盘可靠性、防风控、续传与任务调度**

### 3.2 集成原则

1. Go 后端直接集成官方 `clouddrive.proto`
2. Tauri 只负责静默拉起后端与 CD2，不再承载网盘上传引擎
3. 前端只调用本系统后端 API，不直接访问 CD2
4. 本系统不要求用户打开 CD2 界面完成日常操作

### 3.3 数据原则

1. 不再以 `storageKey / mountPath / rootFolderId / cookie` 作为网盘端点主身份
2. 改为以 `CD2 云账号 + 根路径` 作为端点来源
3. 不再把“上传中间态”作为自研会话持久化的中心
4. 改为持久化“CD2 任务绑定信息 + 本系统展示状态”

## 4. 目标领域模型

## 4.1 CD2 运行时模型

新增运行时模型：

```go
type CD2RuntimeConfig struct {
    BinaryPath      string
    WorkDir         string
    DataDir         string
    GRPCAddress     string
    HTTPAddress     string
    AutoStart       bool
    StartupTimeout  time.Duration
}

type CD2RuntimeState struct {
    Ready           bool
    PID             int
    ProductName     string
    ProductVersion  string
    CloudAPIVersion string
    LastStartedAt   *time.Time
    LastError       string
}
```

职责：

1. 拉起 CD2 进程
2. 健康检查与版本校验
3. 静默后台运行
4. 关闭与重启控制

约束：

1. CD2 必须可由本系统自动拉起
2. 后端启动后优先探测并复用已运行的 CD2
3. 版本不匹配时拒绝进入网盘操作链路

## 4.2 CD2 认证模型

新增认证模型：

```go
type CD2AuthProfile struct {
    Mode            string // password / api_token
    ServerAddress   string
    UserName        string
    TokenRef        string
    TokenExpiresAt  *time.Time
    LastVerifiedAt  *time.Time
}

type CD2ManagedToken struct {
    TokenRef        string
    FriendlyName    string
    RootDir         string
    Permissions     []string
    ExpiresAt       *time.Time
}
```

设计决策：

1. 本系统必须优先使用 **CD2 API Token** 调用网盘能力
2. 用户密码不进入文件操作链路
3. 后端只持有最小权限 Token
4. Token 权限至少覆盖：
   - 文件列表
   - 搜索
   - 创建目录
   - 创建文件/写入
   - 读取
   - 重命名
   - 移动
   - 复制
   - 删除
   - 查看属性
   - 获取传输任务
   - 修改传输任务
   - 获取云 API
   - 修改云 API
   - 推送消息

## 4.3 CD2 云账号模型

新增云账号模型：

```go
type CD2CloudAccount struct {
    CloudName        string
    UserName         string
    NickName         string
    DisplayName      string
    Path             string
    IsLocked         bool
    SupportsUpload   bool
    SupportsDirectURL bool
}

type CD2LoginTicket struct {
    Provider         string
    Mode             string // qrcode / cookie_import
    SessionID        string
    Status           string
    QRCodeText       string
    QRCodeImageURL   string
    CookieRef        string
    CreatedAt        time.Time
    UpdatedAt        time.Time
}
```

设计决策：

1. 115、阿里云盘、百度网盘等都统一归入 `CD2CloudAccount`
2. 115 cookie 仅作为 `cookie_import` 的输入，不进入端点主配置
3. 本系统保存的是“导入工单”和“导入结果”，不是网盘文件操作凭证

## 4.4 存储端点模型

现有 `StorageEndpoint` 保留表结构外壳，但云端点语义重定义。

目标端点类型：

1. `LOCAL`
2. `QNAP_SMB`
3. `REMOVABLE`
4. `CLOUD_DRIVE2`

说明：

1. 不再继续使用 `NETWORK_STORAGE` 作为未来模型主类型
2. `CLOUD_DRIVE2` 明确表达“由 CD2 托管的网盘端点”

`CLOUD_DRIVE2` 端点的目标 `connectionConfig`：

```json
{
  "runtimeId": "default",
  "cloudName": "115",
  "cloudUserName": "xxx",
  "cloudDisplayName": "115 网盘 / xxx",
  "rootPath": "/115/素材库",
  "pathPolicy": "DIRECT_PATH",
  "scanMode": "SUB_FILES",
  "watchMode": "PUSH_THEN_REFRESH",
  "downloadMode": "DIRECT_URL_FIRST",
  "uploadMode": "CD2_CREATE_FILE",
  "deleteMode": "NORMAL",
  "allowContentSearch": false
}
```

字段定义：

1. `runtimeId`：指向当前使用的 CD2 运行时，开发阶段固定为 `default`
2. `cloudName`：CD2 内部云类型，例如 `115`
3. `cloudUserName`：CD2 云账号主键之一
4. `cloudDisplayName`：展示用名称
5. `rootPath`：该端点在 CD2 中可见的根路径
6. `pathPolicy`：固定为直接路径，不依赖挂载点
7. `scanMode`：使用 `GetSubFiles` / `GetSearchResults` 做扫描
8. `watchMode`：使用 `PushMessage` + 主动刷新
9. `downloadMode`：优先直链，失败再回退代理下载
10. `uploadMode`：使用 `CreateFile + WriteToFile(Stream)`
11. `deleteMode`：普通删除，是否永久删除由业务另定

不再允许出现的端点字段：

1. `storageKey`
2. `mountPath`
3. `driver`
4. `rootFolderId`
5. `loginMethod`
6. `appType`
7. `pageSize`
8. `credential`
9. `cookie`

## 4.5 文件与路径模型

新增路径模型：

```go
type CD2PathRef struct {
    EndpointID string
    RootPath   string
    FullPath   string
    Name       string
    IsDir      bool
}

type CD2FileEntry struct {
    Path         string
    Name         string
    Size         int64
    IsDir        bool
    ModifiedAt   *time.Time
    CreatedAt    *time.Time
    PreviewURL   string
    ThumbnailURL string
    CanSearch    bool
    CanMount     bool
}

type CD2DownloadDescriptor struct {
    Path              string
    DirectURL         string
    ProxyPath         string
    UserAgent         string
    AdditionalHeaders map[string]string
    ExpiresIn         *uint64
}
```

设计决策：

1. 扫描与浏览统一基于 `CD2 path`
2. 本系统内部 `Replica.PhysicalPath` 对于云端副本记录 `CD2 FullPath`
3. 不再以 AList mount path 作为物理路径

## 4.6 传输任务模型

新增任务绑定模型：

```go
type CD2TransferBinding struct {
    TaskType        string // upload / download / copy
    ExternalKey     string
    SourcePath      string
    TargetPath      string
    CloudName       string
    CloudUserName   string
}

type CD2TransferSnapshot struct {
    ExternalKey      string
    Status           string
    ProgressPercent  int
    TotalBytes       int64
    FinishedBytes    int64
    BytesPerSecond   float64
    ErrorMessage     string
    OperatorType     string
}
```

设计决策：

1. 本系统不再维护自研的 115 上传会话模型
2. 本系统不再维护 AList/aria2 的外部任务状态模型
3. 只保留“本系统任务记录”与“CD2 任务外键绑定”

以下旧任务元数据将被淘汰：

1. `Aria2`
2. `AList`
3. `Cloud115`
4. `transferEngineKindAria2`
5. `transferEngineKindAList`
6. `transferEngineKindCloud115`

## 4.7 事件模型

新增 CD2 事件模型：

```go
type CD2PushEvent struct {
    EventType       string // transfer / filesystem / mount / log
    ReceivedAt      time.Time
    RawType         string
    RelatedPath     string
    RelatedTaskKey  string
    Message         string
}
```

用途：

1. 任务状态实时刷新
2. 文件变化驱动目录刷新
3. 扫描增量提示

## 5. 模块边界与目录规划

后续新增模块统一放入 `backend/internal/cd2`：

1. `runtime`
   - CD2 进程启动、探活、关闭、版本检查
2. `proto`
   - 官方 proto 生成结果
3. `client`
   - gRPC 连接、认证元数据、通用重试、错误归一化
4. `auth`
   - 登录、Token 管理、权限模板
5. `cloudapi`
   - 云账号接入、账号列表、账号配置
6. `fs`
   - 列表、搜索、详情、目录/文件操作、上传下载
7. `tasks`
   - 上传、下载、复制任务查询与控制
8. `events`
   - `PushTaskChange` / `PushMessage` 消费
9. `mapper`
   - CD2 模型到目录服务模型的转换

目录服务保留的职责：

1. 资产库扫描结果入库
2. 副本、版本、逻辑路径维护
3. 业务任务中心展示
4. 与导入、同步、资产管理页面对接

目录服务不再承担的职责：

1. 网盘协议适配
2. 网盘上传引擎
3. 网盘下载引擎
4. 网盘任务恢复引擎

## 11. 本任务结论

本项目后续的网盘能力改造，不是“给现有网盘抽象换一个实现”，而是：

1. 删除旧网盘执行栈
2. 以 CD2 为中心重建运行时、认证、文件操作和任务桥接
3. 只保留 cookie 获取能力作为 CD2 接入引导工具

该结论自本任务完成后生效。
