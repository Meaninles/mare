Mare 总体架构简介

一、项目定位

Mare 是一款面向个人与小团队的桌面端多媒体资产管理客户端，核心目标是把分散在本地磁盘、NAS、网盘以及可移动存储设备中的图片、视频、音频资产，归并为一个统一、可检索、可恢复、可追踪副本状态的资产库。

项目采用“客户端壳层 + Go 后端服务”的双层架构：

1. Tauri 负责桌面客户端容器能力与前端运行时承载。
2. React + TypeScript 负责桌面客户端界面、交互与状态呈现。
3. Go 负责后台服务、Catalog、连接器、媒体处理、任务编排与本地 HTTP 接口。
4. SQLite 作为本地 Catalog 主存储。

二、产品核心问题

Mare 试图解决以下几类高频问题：

1. 资产分散在多个存储端点，用户无法统一查看。
2. 同一个文件在多个端点上存在多个副本，但缺乏清晰的副本状态认知。
3. 某个端点上的文件被删除、丢失或不可用时，用户难以及时发现并恢复。
4. 图片、视频、音频缺乏统一预览能力，导致资产库可用性不足。
5. 随着资产规模增长，导入、扫描、缩略图、封面、元数据提取等后台任务需要被稳定管理。

三、当前架构原则

Mare 当前架构遵循以下原则：

1. 桌面优先。它不是一个普通网页项目，而是一个桌面客户端程序。
2. Catalog 优先。用户看到的是统一资产，而不是分散文件系统本身。
3. 副本优先建模。资产与副本分离建模，便于后续做恢复、补齐、删除与差异分析。
4. 本地可运行。当前阶段优先保证单机闭环可运行、可测试、可演示。
5. 后端服务化。虽然当前随桌面端本地运行，但后端逻辑按独立 Go 服务组织，便于后续扩展。
6. 连接器抽象。不同存储介质统一通过 Connector 接口接入，减少上层业务对底层协议差异的感知。
7. 媒体处理异步化。缩略图、封面、音频元数据等能力走后台任务，不阻塞主交互。

四、当前系统总体分层

当前代码体现出的主要分层如下：

1. 客户端展示层
   Tauri + React + TypeScript，负责桌面窗口、页面路由、主题切换、资产列表、资产详情、设置、存储管理、同步中心、导入中心等 UI。

2. 客户端服务接入层
   前端通过本地 HTTP 接口读取后端数据，并结合桌面侧能力完成资源访问与交互。

3. Go 应用服务层
   后端在 backend 目录中实现应用启动、配置加载、HTTP 服务、Catalog 服务、Connector 管理、任务记录、媒体处理和日志。

4. Catalog 与任务层
   负责资产、端点、副本、副本版本、任务、媒体预览与音频元数据的组织与写入。

5. 存储接入层
   通过统一 Connector 抽象接入 Local、QNAP、115、Removable 等端点。

6. 本地持久化层
   当前使用 SQLite，已包含核心表与媒体扩展表的 migration。

五、当前已形成的能力轮廓

按现有代码，项目已经具备了以下基础能力：

1. 资产库页面、资产详情页面与设置页面已经形成可用桌面端界面。
2. 已支持浅色与深色两套主题风格。
3. 已具备本地 Catalog 读取、资产列表查询、资产详情查询等基础能力。
4. 已具备端点扫描、全量扫描、单端点重扫的后端入口。
5. 已具备图片缩略图生成、视频封面提取、音频基础元数据读取。
6. 已具备图片、视频、音频基础预览能力。
7. 已具备后台任务记录与任务状态更新能力。
8. 已具备 Go 后端 Air 热重载开发方式。

六、当前 MVP 的真实边界

虽然项目已经实现了不少能力，但整体仍处在 MVP 阶段。当前更强调的是：

1. 打通本地桌面端闭环。
2. 保证 Catalog、扫描、归并、预览、媒体派生物生成链路稳定。
3. 为后续同步恢复、批量补齐、全文搜索、导入规则、配置导入导出等能力预留结构。

七、后续演进方向

Mare 后续主要会沿以下方向推进：

1. 持续完善资产状态与副本状态聚合。
2. 补齐删除、恢复、同步中心与批量补齐工作流。
3. 完善移动设备纳管与导入中心能力。
4. 增强全文搜索、转写搜索与语义搜索。
5. 增强配置导入导出、换机恢复与快速 Catalog 重建。
6. 逐步提升日志、任务中心、错误可解释性和稳定性。
